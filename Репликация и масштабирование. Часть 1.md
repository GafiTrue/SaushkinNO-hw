## Задание 1. Различия master-slave и master-master репликации

**Репликация** — процесс копирования данных с одного сервера БД на другой (или несколько) для повышения отказоустойчивости, масштабирования операций чтения и резервирования данных.

### Master-Slave (ведущий — ведомый)

В этой схеме есть **один главный сервер (master)** и **один или несколько подчинённых (slave)**.

- Все операции **записи** (`INSERT/UPDATE/DELETE`) идут **только на master**.
- Master ведёт **бинарный лог** (binlog) — журнал всех изменений данных.
- Slave подключается к master, скачивает события из binlog в свой **relay log** и проигрывает их у себя, повторяя состояние master.
- Операции **чтения** (`SELECT`) можно распределять между slave-серверами — это и есть масштабирование чтения.

**Плюсы:**
- Простая, хорошо изученная архитектура; минимум конфликтов.
- Лёгкое масштабирование чтения — добавил slave, получил больше read-нагрузки.
- Slave удобно использовать для бэкапов, отчётов и аналитики, не нагружая master.
- Slave работает как «горячий резерв»: при отказе master его можно повысить до нового master.

**Минусы:**
- **Единая точка отказа на запись:** при падении master запись становится недоступной, пока не выполнить failover (вручную или автоматически).
- Между master и slave всегда есть **репликационный лаг** — slave догоняет master с задержкой.
- Чтение со slave может вернуть устаревшие данные (eventual consistency).

### Master-Master (ведущий — ведущий)

В этой схеме **два (или больше) сервера являются master одновременно** — каждый принимает и записи, и чтения, а изменения реплицируются в обе стороны.

- Каждый сервер ведёт свой binlog и одновременно является slave для другого.
- Обычно настраивается с разными `auto_increment_offset` и `auto_increment_increment`, чтобы избежать коллизий по первичным ключам.

**Плюсы:**
- **Отказоустойчивость записи:** при падении одного master клиенты могут писать на второй.
- Возможна гео-распределённая запись (клиенты пишут на ближайший узел).
- Балансировка нагрузки и для чтения, и для записи.

**Минусы:**
- **Конфликты записи:** если оба master одновременно изменили одну и ту же строку — нужно их разрешать, иначе данные разойдутся.
- Сложнее в настройке и эксплуатации, выше требования к мониторингу.
- На практике часто работает как active-passive: запись идёт на один master, второй — резерв.
- Возможен **split-brain** при сбоях сети.

### Краткое сравнение

| Критерий                          | Master-Slave                                  | Master-Master                               |
| --------------------------------- | --------------------------------------------- | ------------------------------------------- |
| Куда пишем                        | Только master                                 | На любой из master                          |
| Куда читаем                       | Master и slave-ы                              | Любой узел                                  |
| Масштабирование чтения            | Да                                            | Да                                          |
| Масштабирование записи            | Нет                                           | Да (с оговорками)                           |
| Отказоустойчивость записи         | Нет (нужен failover)                          | Да                                          |
| Конфликты                         | Практически нет                               | Возможны, нужны механизмы разрешения        |
| Сложность настройки/эксплуатации  | Низкая                                        | Высокая                                     |
| Типичные кейсы                    | Read-heavy приложения, отчётность, бэкапы     | Гео-распределённые системы, HA-кластеры     |

**Вывод:** master-slave — простой и надёжный способ масштабировать чтение и иметь резерв. Master-master даёт отказоустойчивость на запись и возможность писать в нескольких локациях, но платит за это сложностью и риском конфликтов.

## Задание 2. Конфигурация master-slave репликации

Развёрнуто два экземпляра MariaDB 10.11 (MySQL-совместимая) на одной машине, на разных портах и каталогах данных. Master — порт **3307**, slave — порт **3308**.

### 2.1. Структура файлов

```
repl/
├── master/
│   ├── my.cnf          # конфиг master
│   └── data/           # каталог данных master
├── slave/
│   ├── my.cnf          # конфиг slave
│   └── data/           # каталог данных slave
└── logs/
    ├── master.err
    └── slave.err
```

### 2.2. Конфиг master — `master/my.cnf`

```ini
[mysqld]
user            = root
port            = 3307
socket          = /home/claude/repl/master/mysql.sock
pid-file        = /home/claude/repl/master/mysql.pid
datadir         = /home/claude/repl/master/data
log-error       = /home/claude/repl/logs/master.err
server-id       = 1
log_bin         = /home/claude/repl/master/mysql-bin
binlog_format   = ROW
bind-address    = 127.0.0.1
skip-name-resolve
```

Ключевые параметры:
- `server-id = 1` — уникальный ID master;
- `log_bin` — включаем бинарный лог (без него репликация невозможна);
- `binlog_format = ROW` — построчный формат, наиболее безопасный для репликации.

### 2.3. Конфиг slave — `slave/my.cnf`

```ini
[mysqld]
user            = root
port            = 3308
socket          = /home/claude/repl/slave/mysql.sock
pid-file        = /home/claude/repl/slave/mysql.pid
datadir         = /home/claude/repl/slave/data
log-error       = /home/claude/repl/logs/slave.err
server-id       = 2
relay_log       = /home/claude/repl/slave/relay-bin
log_bin         = /home/claude/repl/slave/mysql-bin
binlog_format   = ROW
read_only       = ON
bind-address    = 127.0.0.1
skip-name-resolve
```

Ключевые параметры:
- `server-id = 2` — отличается от master, иначе репликация не запустится;
- `relay_log` — журнал, куда slave пишет события, полученные от master;
- `read_only = ON` — запрет записи на slave для непривилегированных пользователей.

### 2.6. Настройка slave: `CHANGE MASTER TO` и `START SLAVE`

На slave указываем, где master, и стартуем репликацию.

```sql
STOP SLAVE;
CHANGE MASTER TO
  MASTER_HOST     = '127.0.0.1',
  MASTER_PORT     = 3307,
  MASTER_USER     = 'replica',
  MASTER_PASSWORD = 'replicapass',
  MASTER_LOG_FILE = 'mysql-bin.000002',
  MASTER_LOG_POS  = 791;
START SLAVE;
SHOW SLAVE STATUS\G
```

В выводе `SHOW SLAVE STATUS\G` важно увидеть:

- `Slave_IO_Running: Yes` — поток ввода-вывода скачивает binlog;
- `Slave_SQL_Running: Yes` — поток применяет события у себя;
- `Last_Errno: 0`, пустой `Last_Error`;
- `Seconds_Behind_Master: 0` — slave не отстаёт от master.

## Итог

- Master (`127.0.0.1:3307`) принимает запись, ведёт `mysql-bin.*` (binlog).
- Slave (`127.0.0.1:3308`) подключён к master пользователем `replica`, проигрывает binlog в свой relay log и догоняет master в реальном времени (`Seconds_Behind_Master = 0`).
- Запись на slave заблокирована параметром `read_only = ON`.
- Любые изменения, выполненные на master, появляются на slave в течение секунды — репликация работает.
