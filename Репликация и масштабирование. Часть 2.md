## Задание 1. Преимущества разных режимов масштабирования

В обоих случаях речь идёт о **горизонтальном масштабировании на уровне репликации**: добавляем дополнительные серверы БД, чтобы получить отказоустойчивость и/или больше ресурсов для чтения. Разница — в количестве реплик и в том, как они используются.

### 1.1. Активный master + пассивный slave (1 + 1, hot/warm standby)

Один master обслуживает всю нагрузку (и запись, и чтение), а slave работает как «горячий резерв»: получает binlog от master, постоянно догоняет его состояние, но клиенты на него не ходят (или ходят только для бэкапов).

**Основные преимущества:**

- **Высокая доступность (HA).** При отказе master slave можно за считанные минуты (а с менеджером кластера типа Orchestrator / Patroni — за секунды) промоутировать в master, и сервис продолжает работать. Без репликации потеря master = полный простой записи.
- **Снятие бэкапов без влияния на прод.** `mysqldump`, `xtrabackup`, `pg_basebackup` можно запускать на slave — нагрузка ввода-вывода не съест ресурсы у боевого master.
- **Disaster recovery.** Slave обычно ставят в другой стойке / availability zone / датацентре. При полном падении первой локации данные не теряются.
- **Тестирование на «живых» данных.** На slave можно проверять обновления схемы, тяжёлые отчётные запросы, новые версии БД, не задевая клиентов.
- **Простота архитектуры.** Минимум подвижных частей: один master + один slave + механизм репликации. Меньше точек отказа, проще диагностика, проще обучить эксплуатацию.
- **Низкая стоимость.** Всего две машины и никакой дополнительной балансировки чтения — выгодно для проектов, где главное требование «не лежать», а не «выдерживать миллион запросов».

**Где это уместно:** биллинг, ERP, внутренние сервисы — нагрузка умеренная, но простой недопустим.

### 1.2. Master + несколько slave-серверов (1 + N, read scaling)

Один master продолжает быть единственным владельцем записи, но slave-ов становится несколько, и **клиенты читают со slave-ов** (через прокси/драйвер). Каждый slave можно специализировать под свою задачу.

**Основные преимущества:**

- **Линейное масштабирование чтения.** Если 80–95% трафика — это `SELECT` (типичная картина для веб-приложений), добавление каждого нового slave прибавляет реальную пропускную способность по чтению. Master при этом не страдает.
- **Специализация slave-ов.** Один slave обслуживает OLTP-чтение, другой — тяжёлую отчётность/BI, третий — поиск (с другим набором индексов), четвёртый — бэкапы и DR. Master не «забит» аналитическими сканированиями.
- **Гео-распределение.** Slave-ы можно разместить ближе к пользователям (EU/US/APAC) и существенно снизить latency чтения для конечного клиента.
- **Несколько кандидатов на failover.** Если master падает, в master повышается тот slave, который наиболее «свежий». Остальные slave-ы переподписываются на нового master — общий простой минимален.
- **Изоляция нагрузки тяжёлых запросов.** «Тяжёлый» `SELECT` уходит в выделенный аналитический slave и не блокирует ничего на master.
- **Постепенный апгрейд.** Можно сначала обновить версию БД на одном slave, проверить, потом на остальных, потом промоутить — без простоя.

**Где это уместно:** read-heavy веб-сервисы (соц.сети, маркетплейсы, медиа), системы аналитики, поисковые/каталожные сервисы.

### 1.3. Что общего, что разное

| Свойство                       | 1 + 1 (passive slave)         | 1 + N (multiple slaves)              |
| ------------------------------ | ----------------------------- | ------------------------------------ |
| Главная цель                   | Отказоустойчивость + бэкапы   | Масштабирование чтения + HA          |
| Куда идут `SELECT`             | На master (slave — в резерве) | На slave-ы, распределяется по ним    |
| Куда идут `INSERT/UPDATE`      | Только на master              | Только на master                     |
| Сложность эксплуатации         | Низкая                        | Средняя (нужен прокси/балансировщик) |
| Стоимость                      | Низкая                        | Растёт линейно с N                   |
| Масштабирование записи         | Нет                           | Нет (нужен шардинг — см. Задание 2)  |

В обеих схемах **запись остаётся узким местом** — её обслуживает единственный master. Когда упирается уже сама запись, на сцену выходит **шардинг**, к которому переходим в Задании 2.

## Задание 2. План горизонтального и вертикального шардинга

### 2.1. Исходные данные

База данных интернет-магазина книг состоит из трёх таблиц:

- **users** — пользователи: `user_id (PK)`, `email`, `password_hash`, `name`, `country`, `created_at`, …
- **books** — каталог книг: `book_id (PK)`, `title`, `author`, `isbn`, `genre`, `price`, `stock`, …
- **stores** — физические магазины и регионы доставки: `store_id (PK)`, `name`, `region_id`, `address`, `lat`, `lon`, `working_hours`, …

### 2.2. Идея решения — вертикальный шардинг + горизонтальный шардинг

Мы применяем **обе техники одновременно**, потому что они решают разные задачи:

- **Вертикальный шардинг** — это разделение схемы по таблицам/группам таблиц на физически разные кластеры БД. Цель — изолировать рабочие нагрузки и независимо масштабировать каждый домен.
- **Горизонтальный шардинг (sharding)** — это разделение **строк одной и той же таблицы** между несколькими физически разными БД по какому-то ключу (shard key). Цель — обойти ограничение «один master по записи» и масштабировать запись.

### 2.3. Вертикальное разделение

Каждая из трёх таблиц получает **свой собственный кластер БД**:

| Логический кластер | Таблица | Профиль нагрузки                                                |
| ------------------ | ------- | --------------------------------------------------------------- |
| `users-DB`         | users   | Read-heavy на чтение профиля + запись при регистрации/логине    |
| `books-DB`         | books   | Очень read-heavy (каталог, поиск, карточка товара), редкие burst-записи при поступлении товара |
| `stores-DB`        | stores  | Маленький объём, читается часто (геопоиск ближайшего магазина), редкие изменения |

Это даёт:
- независимое масштабирование (если книги растут быстро — наращиваем только `books-DB`);
- изоляцию сбоев (упал каталог книг — авторизация и магазины продолжают работать);
- разные оптимизации (например, для `books-DB` имеет смысл агрессивно кэшировать в Redis перед БД).

### 2.4. Горизонтальное разделение внутри каждого кластера

Внутри каждого вертикального кластера данные шардируются ещё и горизонтально, потому что каждая таблица может вырасти до сотен миллионов строк. Ключ шардирования выбирается так, чтобы **самые частые запросы попадали ровно в один шард** (чтобы не делать дорогих cross-shard JOIN-ов).

| Таблица  | Ключ шардирования  | Стратегия                                         | Почему                                                                                 |
| -------- | ------------------ | ------------------------------------------------- | -------------------------------------------------------------------------------------- |
| users    | `user_id`          | hash / modulo: `shard = user_id % N`              | Равномерное распределение, типовой запрос — «логин по id/email» попадает в один шард   |
| books    | `book_id`          | range-based: id 1…1M → shard A, 1M…2M → shard B   | Каталог хорошо делится по диапазонам, легко добавлять новые шарды для новых поступлений |
| stores   | `region_id`        | directory / geo-based: EU → shard 1, US → shard 2 | Запросы «магазины рядом со мной» по природе локальны — идут в один регион              |

### 2.5. Уровни архитектуры (сверху вниз)

1. **Приложение (backend)** — не знает про физическое размещение шардов, общается только с прокси.
2. **Router / Proxy** (ProxySQL, HAProxy, Vitess или прикладной шард-роутер) — принимает запросы, по ключу шардирования (`user_id`, `book_id`, `region_id`) определяет, в какой кластер их направить, и отделяет `SELECT` от `INSERT/UPDATE`.
3. **Кластеры шардов.** Каждый шард — это **master-slave кластер**: 1 master на запись + 1–2 slave на чтение. Внутри каждого шарда работает обычная репликация из Задания 1.

### 2.6. Блок-схема итоговой архитектуры

<img width="1739" height="791" alt="image" src="https://github.com/user-attachments/assets/9dd5cce4-6178-4a1d-9b06-a72d8d840c7a" />

**Расшифровка диаграммы:**
- Жёлтый блок — `users-DB`, разбит на 2 горизонтальных шарда по `user_id % 2`.
- Синий блок — `books-DB`, разбит на 2 шарда по диапазону `book_id`.
- Зелёный блок — `stores-DB`, разбит на 2 шарда по `region_id` (EU и US).
- В каждом шарде: красный цилиндр — **master** (`rw`), зелёные цилиндры — **slave** (`ro`); пунктирные стрелки — поток репликации (binlog/relay log).
- Сверху единый Router/Proxy решает, в какой шард уйдёт запрос, по ключу шардирования.

### 2.7. Режимы работы серверов

| Сервер                               | Роль   | Режим              | Что делает                                          |
| ------------------------------------ | ------ | ------------------ | --------------------------------------------------- |
| master любого шарда                  | MASTER | `read_write`       | Принимает `INSERT/UPDATE/DELETE` и любые `SELECT`; пишет binlog |
| slave любого шарда                   | SLAVE  | `read_only = ON`   | Получает binlog с master, проигрывает его в relay log, отдаёт `SELECT` через прокси |
| дополнительный slave (опционально)   | SLAVE  | `read_only = ON`   | Используется только для бэкапов и аналитических отчётов, чтобы не «тревожить» OLTP-slave |
| Router / Proxy                       | PROXY  | без хранения данных | Маршрутизация по shard key, разделение `r/w` потоков, мониторинг здоровья шардов, failover |

**Сценарий записи:** клиент → Proxy → определяется ключ шардирования → выбирается master нужного шарда → выполняется запись → master реплицирует изменения на свои slave.

**Сценарий чтения:** клиент → Proxy → определяется шард по ключу → запрос балансируется на один из живых slave этого шарда (или на master, если нужна строгая консистентность read-after-write).

**Сценарий отказа master шарда:** Proxy (или менеджер кластера) обнаруживает недоступность master → промоутит наиболее «свежий» slave этого шарда → остальные slave-ы переключают репликацию на нового master → запись возобновляется. Остальные шарды и кластеры всё это время работают как ни в чём не бывало — отказ изолирован.

### 2.8. Что и зачем

- **Вертикальный шардинг** дал нам три независимых отказоустойчивых домена, каждый со своим жизненным циклом и нагрузкой.
- **Горизонтальный шардинг** внутри каждого домена снял главное ограничение master-slave — потолок по записи: теперь запись горизонтально масштабируется добавлением новых шардов.
- **Master-slave внутри каждого шарда** обеспечивает HA и масштабирование чтения внутри одного шарда.
- **Прокси** скрывает всю эту сложность от приложения и реализует политику маршрутизации и failover.
