### Задание 1 СУБД

Крупная строительная компания, которая также занимается проектированием и девелопментом, решила создать правильную архитектуру для работы с данными. Ниже представлены задачи, которые необходимо решить для каждой предметной области.

Какие типы СУБД, на ваш взгляд, лучше всего подойдут для решения этих задач и почему?

1.1. Бюджетирование проектов с дальнейшим формированием финансовых аналитических отчётов и прогнозирования рисков. СУБД должна гарантировать целостность и чёткую структуру данных. \
Реляционные СУБД(SQL), PostgreSQL потому что нужна четка структура и SQL дает возможность гибко работать с данными \
1.2. Под каждый девелоперский проект создаётся отдельный лендинг, и все данные по лидам стекаются в CRM к маркетологам и менеджерам по продажам. Какой тип СУБД лучше использовать для лендингов и для CRM? СУБД должны быть гибкими и быстрыми. \
Для лендингов окументо-ориентированная СУБД (NoSQL), например MongoDB, т.к подобные данные обычно имеют разный набор полей. Для CRM если СУБД должно вести историю сделок и финансов, то лучше подойдет реляционная, если для быстро взоимодействия, то опять же MongoDB \
1.3. Отдел контроля качества решил создать базу по корпоративным нормам и правилам, обучающему материалу и так далее, сформированную согласно структуре компании. СУБД должна иметь простую и понятную структуру. \
Реляционная СУБД, например SQLite или PostgreSQL \
1.4. Департамент логистики нуждается в решении задач по быстрому формированию маршрутов доставки материалов по объектам и распределению курьеров по маршрутам с доставкой документов. СУБД должна уметь быстро работать со связями. \
Графовые СУБД(Graph DB). Задача поиска кратчайшего пути, распределения курьеров по точкам — это классические задачи теории графов. В графовых БД сущности (объекты, склады) — это узлы (nodes), а дороги и маршруты — это связи (relationships). \

### Задание 2 Транзакции

2.1. Пользователь пополняет баланс счёта телефона, распишите пошагово, какие действия должны произойти для того, чтобы транзакция завершилась успешно. Ориентируйтесь на шесть действий. \

1. Проверка входных данных: Система проверяет, существует ли указанный номер телефона, корректен ли счет списания (карта/счет клиента) и достаточно ли на нем средств. \
2. Блокировка (Lock) ресурсов: Система блокирует запись о счете плательщика для изменений из других параллельных процессов (чтобы не списать деньги дважды). \
3. Списание средств (Debit): Уменьшается баланс на счете плательщика на сумму пополнения. \
4. Зачисление средств (Credit): Увеличивается баланс на лицевом счете телефона (или счете телеком-оператора). \
5. Фиксация (Commit): Приложение отдает команду COMMIT. Если оба изменения (списание и зачисление) прошли успешно, они фиксируются в базе данных окончательно. \
6. Завершение и уведомление: Транзакция завершена, блокировки сняты. Пользователь получает уведомление об успешном пополнении. \

### Задание 3 SQL vs NoSQL
3.1. Пять преимуществ SQL-систем \
1. ACID-транзакции (Надежность): Гарантируют, что транзакция будет выполнена полностью или отменена целиком. Это критично для финансов, бронирований и любых систем, где важна целостность данных. \
2. Структура и консистентность: Четкая схема данных (типы полей, ограничения) предотвращает ошибки приложения и гарантирует, что в базе не окажется "мусора" \
3. Стандартизированный язык (SQL): Огромное сообщество, тонны обучающих материалов и инструментов. Разработчики могут переходить между разными SQL-базами (MySQL, PostgreSQL) с минимальной адаптацией. \
4. Связи между данными (Relationships): Благодаря системе внешних ключей (Foreign Keys) данные нормализованы, что исключает дублирование и противоречия (например, адрес клиента хранится в одном месте, а не в тысяче заказов). \
5. Мощные инструменты аналитики: Поддержка сложных JOIN'ов, подзапросов, оконных функций и представлений (views) позволяет строить сложнейшие отчеты "из коробки". \

### Задание 4  Кластеры

Критерий выбора: \
При наличии 1000 машин и необходимости "большого количества вычислений над огромным количеством данных", ключевым критерием будет модель распределенных вычислений (параллельной обработки) . \

Какая модель справится лучше? \
Лучше всего справятся СУБД, использующие архитектуру MPP (Massively Parallel Processing) . \

Почему MPP? \
В MPP-системах (например, Greenplum, Vertica, Amazon Redshift, ClickHouse) данные физически разрезаны (шардированы) по всем 1000 машинам кластера. Каждая машина имеет свой процессор, память и диск. \
1. Распараллеливание запроса: Когда приходит сложный аналитический запрос (например, "посчитать средний чек за 5 лет"), координатор разбивает его на 1000 маленьких подзапросов и отправляет каждую машину. \
2. Локальность данных: Каждая машина читает ТОЛЬКО СВОИ кусок данных с диска и вычисляет локальный результат. \
3. Агрегация: Результаты (1000 частичных сумм) отправляются обратно координатору, который складывает их и выдает финальный ответ. \
Какой тип СУБД выбрать? \
Для аналитики (OLAP): Столбцовые (columnar) MPP-СУБД, такие как ClickHouse или Apache Druid. Они оптимизированы для чтения огромных объемов данных и быстрых агрегаций. \
Для хранения и обработки "сырых" данных: Платформы типа Hadoop (HDFS + Hive/Spark) . Они позволяют хранить данные в неструктурированном виде и обрабатывать их параллельно на всех узлах кластера, используя модели MapReduce или Spark. \

Итог: Для 1000 машин нужна СУБД с горизонтальной масштабируемостью и поддержкой MPP-архитектуры, чтобы заставить каждую машину параллельно считать свой кусок данных.
