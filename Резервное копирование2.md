## Задание 1. Резервное копирование

### 1.1. Восстановление данных в полном объёме за предыдущий день

В этом случае подойдёт **полное (full) резервное копирование**, выполняемое ежедневно по расписанию (например, ночью, в окне минимальной нагрузки).

Стратегия:

* Раз в сутки снимается полный бэкап всей БД и кладётся в защищённое хранилище (отдельный сервер, S3-совместимое хранилище, ленточный накопитель).
* Хранится несколько последних полных копий (например, ротация на 7–14 дней).
* При сбое восстановление происходит из последнего ночного бэкапа, что даёт RPO ≈ 24 часа.

Альтернативный вариант для крупных БД, где полный ежедневный бэкап слишком дорог по времени и месту: **схема full + differential**, когда раз в неделю снимается полный бэкап, а в остальные дни только дифференциальные относительно последнего полного. Для восстановления нужно применить full + последний differential.

### 1.2. Восстановление данных за час до предполагаемой поломки

Здесь обычного ежедневного бэкапа недостаточно, потому что между бэкапами теряется до суток данных. Нужен механизм **Point-in-Time Recovery (PITR)**, который собирается из двух компонентов:

1. **Базовый полный бэкап** (раз в сутки или реже).
2. **Непрерывное архивирование журналов транзакций**:
   * для PostgreSQL — WAL-сегменты (Write-Ahead Log) через `archive_command` или `pg_receivewal`;
   * для MySQL — binary log (`log_bin`).

Восстановление: разворачивается ближайший по времени полный бэкап, а затем поверх него «накатываются» журналы транзакций до нужной точки во времени (например, `recovery_target_time = '2026-05-12 14:00:00'`). Это позволяет получить состояние БД с точностью до отдельной транзакции и обеспечить RPO в минутах или даже секундах.

Дополнительно можно использовать **инкрементальные бэкапы** (pgBackRest, Percona XtraBackup), чтобы уменьшить объём WAL/binlog, которые надо проигрывать при восстановлении.

### 1.3.* Моментальное переключение на работающую БД при поломке

Да, такой кейс реализуем и широко применяется в финансовых системах, где простой даже на минуты неприемлем. Это уже задача не резервного копирования, а **High Availability (HA)**. Основные подходы:

* **Репликация master / standby с автоматическим failover.** Hot standby постоянно получает поток изменений с мастера (синхронно или асинхронно) и в случае сбоя промоутится в новый мастер. В PostgreSQL это типично реализуется через **Patroni + etcd/Consul + HAProxy**; в MySQL — через **MHA**, **Orchestrator** или **MySQL Group Replication**.
* **Синхронные multi-master кластеры.** Например, **Galera Cluster** для MySQL/MariaDB или **PostgreSQL BDR** — все узлы принимают запись, данные синхронны, при выпадении одного узла остальные продолжают работу прозрачно для клиента.
* **Shared storage HA** на уровне ОС (Pacemaker + Corosync + DRBD): отдельный экземпляр БД на резервном узле поднимается на том же дисковом томе при падении основного.
* **Облачные managed-решения** (Amazon RDS Multi-AZ, Google Cloud SQL HA, Azure Database for PostgreSQL Flexible Server) предоставляют автоматический failover «из коробки» с переключением за десятки секунд.

Важно понимать, что HA-кластер **не отменяет резервное копирование**: репликация защищает от аппаратных сбоев, но не от логических ошибок (удалили не ту таблицу, испорчены данные приложением, ransomware). Логическая ошибка моментально реплицируется на все узлы. Поэтому в проде всегда сочетают HA + регулярные бэкапы с PITR.

## Задание 2. PostgreSQL

### 2.1. Пример команды резервирования и восстановления БД (pg_dump / pg_restore)

Согласно официальной документации PostgreSQL ([pg_dump](https://www.postgresql.org/docs/current/app-pgdump.html), [pg_restore](https://www.postgresql.org/docs/current/app-pgrestore.html)).

**Создание бэкапа в custom-формате** (рекомендуемый формат, так как поддерживает выборочное восстановление и сжатие):

```bash
pg_dump -U postgres -h localhost -p 5432 \
        -F c -b -v \
        -f /var/backups/mydb_$(date +%Y%m%d).dump \
        mydb
```

Расшифровка флагов:

* `-U postgres` — имя пользователя БД.
* `-h localhost` — хост, на котором запущен PostgreSQL.
* `-p 5432` — порт.
* `-F c` — формат custom (бинарный, со сжатием, пригоден для `pg_restore`).
* `-b` — включить large objects.
* `-v` — verbose-вывод процесса.
* `-f` — путь к файлу бэкапа.
* `mydb` — имя БД для резервирования.

**Восстановление из custom-бэкапа:**

```bash
pg_restore -U postgres -h localhost -p 5432 \
           -d mydb_restored \
           -v --clean --if-exists \
           /var/backups/mydb_20260512.dump
```

Где:

* `-d mydb_restored` — целевая БД (должна быть предварительно создана через `createdb`).
* `--clean --if-exists` — перед восстановлением удаляются существующие объекты (полезно при перезаливке).

Для бэкапа **всех баз кластера** используется `pg_dumpall`:

```bash
pg_dumpall -U postgres -h localhost -f /var/backups/all_$(date +%Y%m%d).sql
```

### 2.1.* Автоматизация процесса

Да, процесс легко автоматизируется. Несколько уровней автоматизации, от простого к сложному:

**1. cron + bash-скрипт.** Минимальный рабочий вариант для маленьких проектов:

```bash
# /etc/cron.d/pg_backup
0 2 * * * postgres /usr/local/bin/pg_backup.sh
```

В самом скрипте: `pg_dump`, ротация старых файлов (`find /var/backups -mtime +14 -delete`), отправка результата в S3 через `aws s3 cp`, уведомление в Slack/Telegram при ошибке.

**2. systemd-таймеры** вместо cron, если хочется лучшего логирования через `journalctl` и зависимостей между юнитами.

**3. Специализированные инструменты** для продакшна:

* **pgBackRest** — продвинутый бэкап-менеджер с поддержкой параллельного бэкапа, дельта-бэкапов, шифрования, PITR, заливки в S3/GCS/Azure из коробки.
* **Barman (Backup and Recovery Manager for PostgreSQL)** — управление множеством PostgreSQL-серверов с одного бэкап-узла, retention policy, проверка целостности.
* **WAL-G / WAL-E** — потоковая отправка WAL и базовых бэкапов в облачное объектное хранилище.

**4. Kubernetes-операторы** (Zalando Postgres Operator, CloudNativePG, Crunchy Postgres Operator) — там бэкапы декларативно описываются в манифесте и выполняются оператором.

**5. Managed-сервисы** (RDS, Cloud SQL, Supabase) делают всё это автоматически: достаточно указать retention и окно бэкапа.

Что бы ни использовалось, важна периодическая **проверка восстановления** на отдельном стенде. Бэкап, который никогда не восстанавливали, считается несуществующим.

## Задание 3. MySQL

### 3.1. Пример команды инкрементного резервного копирования базы данных MySQL

В MySQL инкрементальный бэкап классически реализуется через **binary log** ([документация MySQL: Point-in-Time Recovery Using Binary Log](https://dev.mysql.com/doc/refman/8.0/en/point-in-time-recovery.html)).

**Шаг 1. Включить binary logging** в конфиге `/etc/mysql/my.cnf`:

```ini
[mysqld]
server-id = 1
log_bin = /var/log/mysql/mysql-bin
binlog_format = ROW
expire_logs_days = 14
```

**Шаг 2. Сделать полный бэкап** (с фиксацией позиции в binlog):

```bash
mysqldump -u root -p \
          --all-databases \
          --single-transaction \
          --flush-logs \
          --master-data=2 \
          --triggers --routines --events \
          > /var/backups/mysql/full_$(date +%Y%m%d).sql
```

Ключевые опции:

* `--single-transaction` — консистентный снимок для InnoDB без блокировок.
* `--flush-logs` — закрыть текущий binary log и начать новый, чтобы инкрементальные бэкапы шли с чистой границы.
* `--master-data=2` — записать координаты binlog (имя файла + позицию) в дамп комментарием.

**Шаг 3. Инкрементальный бэкап** — это просто архивирование binary log файлов, появившихся после полного бэкапа:

```bash
# Принудительно закрыть текущий binlog
mysqladmin -u root -p flush-logs

# Скопировать накопленные binlog в хранилище
cp /var/log/mysql/mysql-bin.0000* /var/backups/mysql/incremental/
```

**Восстановление до точки во времени:**

```bash
# 1. Восстановить полный бэкап
mysql -u root -p < /var/backups/mysql/full_20260512.sql

# 2. Применить binary logs до нужного момента
mysqlbinlog --stop-datetime="2026-05-12 14:00:00" \
            /var/backups/mysql/incremental/mysql-bin.000123 \
            /var/backups/mysql/incremental/mysql-bin.000124 \
   | mysql -u root -p
```

**Альтернатива: Percona XtraBackup** ([документация](https://docs.percona.com/percona-xtrabackup/8.0/index.html)). Поддерживает физические инкрементальные бэкапы без блокировок:

```bash
# Полный бэкап
xtrabackup --backup --target-dir=/var/backups/mysql/full \
           --user=root --password=...

# Инкрементальный относительно полного
xtrabackup --backup --target-dir=/var/backups/mysql/inc1 \
           --incremental-basedir=/var/backups/mysql/full \
           --user=root --password=...

# Следующий инкрементальный — относительно предыдущего
xtrabackup --backup --target-dir=/var/backups/mysql/inc2 \
           --incremental-basedir=/var/backups/mysql/inc1 \
           --user=root --password=...
```

### 3.1.* Когда реплика даёт преимущество по сравнению с обычным резервным копированием

Реплика и бэкап решают разные задачи, но в ряде сценариев реплика существенно выигрывает:

* **Минимизация влияния бэкапа на продакшн.** `mysqldump` и даже `xtrabackup` на горячей мастер-БД создают I/O-нагрузку и могут притормаживать пользовательские запросы. Если снимать бэкап с реплики, мастер не трогаем вообще.
* **RPO близкое к нулю.** Резервная копия отстаёт от мастера на интервал между бэкапами (часы или сутки). Асинхронная реплика отстаёт на секунды, синхронная — догоняет в режиме реального времени. При сбое мастера потери данных минимальны.
* **RTO в секунды.** Восстановление из бэкапа большой БД (сотни ГБ или терабайты) занимает часы. Переключение на готовую реплику с уже накатанными данными занимает секунды (промоутинг slave в master).
* **Распределение read-нагрузки.** Реплика одновременно выполняет роль read-only узла для аналитических и тяжёлых SELECT-запросов, разгружая мастер. Бэкап такой пользы не приносит.
* **Географическое резервирование (disaster recovery).** Реплика в другом дата-центре защищает от катастрофы целого ЦОДа. Бэкап в том же ЦОДе от такого не спасёт.
* **Тестовые и аналитические среды.** На основе реплики можно делать снапшоты для разработчиков и аналитиков, не нагружая прод.
* **Проверка консистентности.** На реплике удобно периодически прогонять `pt-table-checksum` и убеждаться, что данные не разъехались.

Где реплика **проигрывает** бэкапу и почему её одной недостаточно:

* Логические ошибки (DROP TABLE, UPDATE без WHERE, действия зловреда) **мгновенно реплицируются** на все узлы. Бэкап с retention в 30 дней спасёт, реплика — нет.
* Реплика не защищает от повреждения данных, вызванного багом в приложении или MySQL.
* Бэкап позволяет откатиться на точную точку в прошлом (PITR), реплика всегда отражает текущее состояние.

Поэтому промышленные конфигурации сочетают оба подхода: репликация для HA и распределения нагрузки + регулярные бэкапы с PITR для защиты от логических ошибок и долговременного хранения.
